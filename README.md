## О проекте

Покупателю сложно объективно оценить товар: один и тот же товар продаётся разными продавцами и присутствует на разных площадках, отзывы разрознены и не сведены воедино, а часть из них недостоверна из-за накруток.

**ReviewMonitoring** решает эту задачу. По ссылке на товар система автоматически:

1. собирает данные о товаре и отзывах с маркетплейса;
2. находит то же предложение у других продавцов и на других площадках;
3. сводит всё воедино и формирует аналитику — в двух режимах:
   - **статистический** — детерминированные количественные показатели (средневзвешенная оценка, распределение оценок, диапазон цен, лучшее предложение);
   - **интеллектуальный** — анализ содержания отзывов большой языковой моделью: достоинства, недостатки, ключевые слова, признаки недобросовестных продавцов и связное обобщение.

Проект разработан как выпускная квалификационная работа магистратуры и доведён до рабочего состояния с акцентом на чистую архитектуру, расширяемость и устойчивость.

## Возможности

- **Кросс-платформенный сбор** — единый сбор данных с Ozon и Wildberries через их внутренние программные интерфейсы.
- **Сопоставление одного товара у разных продавцов** — поиск того же товара на разных площадках с генерацией поисковых запросов через LLM и интеллектуальным отсевом нерелевантных предложений.
- **Два режима анализа** — быстрый статистический (без внешних сервисов) и глубокий интеллектуальный (через языковую модель).
- **Выявление накруток** — сопоставление заявленных метрик (рейтинг, число отзывов) с содержанием отзывов: расхождение помечается как тревожный признак.
- **Асинхронная обработка** — запрос возвращает идентификатор задачи мгновенно, обработка идёт в фоне, состояние отслеживается по идентификатору (в том числе потоковой передачей).
- **Устойчивость к сбоям** — при недоступности площадки обработка продолжается по доступным источникам (частичный результат); при ошибке языковой модели выполняется откат к статистическому анализу.
- **Демо-режим** — запуск и демонстрация без внешней инфраструктуры (in-memory кэш, заглушки провайдеров и LLM).

## Архитектура

Проект построен по принципам **Clean Architecture** — разделение на слои с однонаправленными зависимостями, направленными внутрь, к доменному ядру. Бизнес-логика не зависит от внешних деталей (базы данных, площадок, языковой модели), что обеспечивает расширяемость и тестируемость.

```
┌─────────────────────────────────────────────────────────────┐
│                      Presentation (API)                       │
│            ASP.NET Core · приём запроса · стриминг            │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│                        Application                            │
│     сценарии (MediatR) · оркестратор · интерфейсы            │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│         Infrastructure          │          Ingestion          │
│   PostgreSQL · Redis · LLM      │   провайдеры площадок        │
│   (реализации интерфейсов)      │   (Ozon, Wildberries)        │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│                          Domain                               │
│        сущности · модель задачи · правила (ядро)             │
└─────────────────────────────────────────────────────────────┘
```

### Ключевые архитектурные решения

- **Провайдеры площадок на едином интерфейсе** (`IIngestionProvider`) с динамическим подключением через рефлексию — новая площадка добавляется реализацией интерфейса, без изменения остального кода.
- **Асинхронный конвейер** — постановка задачи в очередь на `System.Threading.Channels`, координация этапов компонентом-оркестратором (`JobProcessor`), параллельная обработка провайдеров через `Task.WhenAll`.
- **Паттерн «медиатор»** (MediatR) — операции оформлены как запросы/команды с обработчиками, что снижает связанность слоёв.
- **Разделение хранения по времени жизни** — состояние выполняющихся задач в Redis (быстрый доступ, TTL), завершённые задачи с результатом в PostgreSQL (в том числе сложные структуры в `jsonb`).
- **Инкапсуляция доступа к LLM** в компоненте `AiClient` через агрегатор OpenRouter (OpenAI-совместимый API) — конкретная модель выбирается через конфигурацию.

## Как это работает

```
POST /api/analysis
  │   { "query": "<ссылка на товар>", "mode": "LLM", "config": { ... } }
  │
  ▼
создаётся задача → возвращается jobId  ◄── мгновенно, без ожидания
  │
  ▼  (фоновая обработка)
сбор данных ──► поиск у других продавцов ──► отсев нерелевантных (LLM)
  │                                                      │
  ▼                                                      ▼
агрегация статистики ──► интеллектуальный анализ (LLM) ──► результат
  │
  ▼
GET /api/jobs/{jobId} → состояние / итоговый результат
```

Состояние задачи доступно по `jobId` в любой момент — опросом или потоковой передачей обновлений.

## Технические детали, которыми горжусь

- **Работа с внутренними API площадок** — решены реальные проблемы доступа: подбор заголовков, работа с cookie-сессией, обход защиты от автоматизации, gzip-декомпрессия ответов, обращение к нескольким узлам отзывов у Wildberries.
- **Агрегация готовых распределений** — статистика вычисляется сложением распределений оценок, полученных от площадок, а не пересчётом из отдельных отзывов: это даёт точные данные там, где они доступны, и независимость от объёма собранной выборки.
- **Batch-анализ отзывов за один запрос к LLM** — каждому предложению присваивается индекс, по которому результаты сопоставляются с предложениями; предварительное сэмплирование отзывов сохраняет представленность разных оценок.
- **Graceful degradation** — система не падает при отказе площадки или ошибке модели, а возвращает корректный частичный/статистический результат.

## Запуск

> Требуется .NET 8 SDK. Для полного режима — PostgreSQL и Redis; для демонстрации достаточно демо-режима.

```bash
git clone https://github.com/dcobol78/ReviewMonitoring.git
cd ReviewMonitoring

# восстановление зависимостей и сборка
dotnet restore
dotnet build

# запуск API
dotnet run --project src/ReviewMonitoring.Api
```

Конфигурация (строки подключения к PostgreSQL и Redis, ключ доступа к OpenRouter, данные сессии площадок) задаётся через `appsettings` / User Secrets и не хранится в репозитории.

**Демо-режим** (`DemoMode`) позволяет запустить и проверить полный цикл обработки без внешней инфраструктуры — на in-memory кэше, заглушках провайдеров и LLM.
